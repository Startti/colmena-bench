The cost problem

Every turn, other frameworks re-pay for the whole conversation

Mainstream frameworks resend the entire growing conversation — and every large tool output — on each turn. Colmena keeps context lean by design: documents and bulky tool results are managed by the engine, not pinned into history.

A 10-turn session costs about $0.018 instead of $0.13–$0.14 — same model, same answers. The longer the conversation, the wider the gap.

One competitor did close the gap — after we hand-wrote 15–20 lines of scrubbing code per tool into Google ADK. That is the point: what Colmena's engine does for every agent by default, others reach only with code your team must write, test, and maintain — per tool, per agent, forever.

The trust problem

Customer credentials should never reach the model

When an agent collects an API key or password mid-conversation, every framework we tested — using its normal, recommended pattern — placed that secret straight into the model's transcript. Colmena encrypts it to an opaque handle and injects the real value only into the outbound call.

Tested on two attack surfaces (collecting the secret, and a tool echoing it back), three seeds each — six cells per framework, audited at the proxy. Every framework still delivers the real credential to the endpoint, 6/6 — Colmena does the same work, it just never exposes the value to the model.

One hand-built arm also reaches 0%: LangGraph, after we architected the credential path out of the model's reach with interrupt() and a checkpointer. Safe by default on one side; a per-agent construction task on the other.

The maintenance problem

Production safety as configuration, not code you maintain

On a refund-authorization task with a human approval gate, an automatic critic-retry loop, credential masking, and a $100 escalation policy, all six frameworks pass all four criteria. The capability is not the difference — the authoring model is. Competitors carry their hardening entirely as imperative code your team writes, tests, and keeps working; the bulk of Colmena's is declarative configuration the engine interprets.

Authoring split for the same passing solution. Colmena's is not the smallest — 235 lines in total — and we claim no line-count advantage. The difference is the kind: change the approval policy or the retry gate in Colmena and you edit a document the same engine re-interprets; in the others you edit, re-test, and redeploy code.
A passing run proves the hand-rolled code was written correctly for that agent, on that run. The engine invariant is what guarantees the same behavior for the next agent your team ships.

The deployment problem

An agent is a file you hand the server — not a code deploy

This is the sharpest architectural difference. In the other frameworks an agent is a program: to add a new agent or change one, your engineers edit code and redeploy the service. In Colmena an agent is a configuration file a generic server runs — one platform serves many agents, and you add, version, or roll one back by handing it a new file. No code change, no redeploy.

Colmena — configuration 
// agent.json — the agent IS this file
{
  "nodes": {
    "classify": { "type": "llm_call", ... },
    "lookup":   { "type": "http_request" },
    "reply":    { "type": "llm_call" }
  },
  "edges": [
    { "from": "classify", "to": "lookup" },
    { "from": "lookup",   "to": "reply" }
  ]
}
Hand this document to the running server → it runs. Change it → it changes.
Other frameworks — code 
# agent.py — the agent IS code
from langgraph.graph import StateGraph
def classify(state): ...
def lookup(state): ...
def reply(state): ...

g.add_node("classify", classify)
g.add_edge("classify", "lookup")
g.add_edge("lookup", "reply")
# ...then rebuild & redeploy
Edit the program, rebuild, redeploy to ship a change.
Honest framing: this is an architecture and operating-model advantage, not a benchmark number, and not a "fewer lines of code" claim. Another team could build a configuration layer over a Python framework — but that layer is exactly what Colmena already is.

Why you can trust these numbers

Independent meteringTokens and cost captured by a proxy between every framework and the AI provider — read from the traffic that crossed the wire, never self-reported.
Identical, idiomatic conditionsSame model (gemini-2.5-flash), same temperature, same inputs and driver scripts; each rival built the way its own documentation recommends — no handicaps.
Quality-gatedEvery comparison first passes the same answer-quality check against fixed ground truth. No framework saves tokens by giving a worse answer.
Steelmen includedWhere a competitor can match Colmena with hand-written code, we wrote that code and report it — including where it beats Colmena on tokens (28,795 vs 39,085).
Trade-offs, stated plainly: Colmena is not faster per session. Per-call latency is at parity with the fastest competitor (median 761 ms vs 738 ms), but the same lazy loading and compaction that buy the token savings cost extra model round-trips — roughly 18 per session against 13. A deployment bound by wall-clock latency rather than token cost should weigh that trade. And every safeguard measured here is reachable in a competitor by hand — the paper's claim is who does the work: an engine invariant your team cannot omit, or code it must remember to write, correctly, every time.
