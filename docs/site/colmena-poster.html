Provider-authoritative agent-framework benchmark

A Provider-Authoritative Benchmark of LLM Agent Frameworks: Context Cost, Credential Isolation, and Agents as Configuration

Colmena Engineering · benchmark suite colmena-bench
We compare Colmena (Rust) against CrewAI, LangChain, LangGraph, LlamaIndex, and Google ADK on identical tasks, metering every token at a shared LiteLLM proxy. Colmena uses 10–12× fewer input tokens on multi-turn work, leaks credentials to the model in 0% of runs (vs 100% for every competitor), and ships production-safety controls as declarative config. We report the advantages — and the trade-offs — under one reproducible harness.

1 Motivation

Framework benchmarks are usually self-reported

Agent frameworks are typically measured by their own authors, who control what is counted. We built a neutral harness where every LLM call for all six frameworks passes through one proxy that records the authoritative token and cost figures.

The question: holding the model, prompts, and tasks fixed, where does the framework itself change cost, safety, and the code you maintain?

2 Method

One proxy, identical conditions

  • Token authority. A LiteLLM proxy sits between every framework and Google AI; per-run spans give provider-authoritative input/output tokens and USD.
  • Competitor attribution. Each Python run is tagged with an x-bench-run-id header so its spans are isolated.
  • Colmena measurement. Colmena cannot send that header, so its tokens are read as a line-count delta of the proxy session file — runs are serialized to keep the delta attributable.
  • Fairness. Same model and temperature 0 for all; each competitor used with its own default memory and idioms; framework versions pinned.

4 Result · the context tax

Colmena stays flat while others spike on every chart turn

Colmena 39,085 vs 404,095–452,359 input tokens; $0.018 vs $0.125–$0.142 per 10-turn session. Answer-quality pass-rate 1.00 across all six. Mechanism: ephemeral attachments + a binary tool-output scrubber keep history from accumulating — the paper's default-versus-hand-written-code thesis.

5 Result · tools at scale

Lazy tool loading scales with catalog size

With many tools, competitors resend every tool schema each turn. Colmena sends a compact catalog and fetches a schema on demand.

10-turn / 30-tool, cumulative66,808 vs 111k–125k
200-tool single-shot probe22,190 vs 45k–104k
Tool-selection accuracy1.00 across all

Attribution: most of the multi-turn gain is conversation-memory compaction (helps lazy and eager alike); lazy loading is the clean differentiator as tool count grows. The advantage is cost, not accuracy.

All six frameworks pass all four criteria on the refund task — the capability is not the gap. 9 Architecture · agents as data

Configuration, not code

Add / change an agent
ColmenaPOST a new JSON graph · no redeploy
Python frameworksedit code · rebuild · redeploy

In production (API + worker fleet) the agent is a configuration document a generic server interprets — one server serves many agents. The competitors author agents in code. This is an operating-model difference, not a benchmark number, and not a line-count claim; a DIY config layer over a Python framework is exactly what Colmena already ships.

Trade-offs and limitations

  • Not faster per session, no in-process parallelism — by design Colmena scales as many distinct agents on one engine, not one agent at high throughput; ~18 vs 13 LLM calls on the Context Tax.
  • No per-token price advantage — same model, same price; the advantage is context volume.
  • On trivial agents, the code-size advantage disappears.
  • The Query-Strategy Trade-off: the same compaction that drives the Context Tax result costs up to ~7 pts of accuracy on large mid-conversation tables (Colmena expert 93–97% vs 97–100%); tunable.
